# 1. 데이터베이스 모델링

모델링: 모형을 만드는 일

## DataBase Modeling

<img src="./img/1.png">

- 현실 세계의 데이터를 체계적으로 분석하여
- 어떤 데이터가 있고, 서로 어떻게 연결되는지 설계(데이터의 논리적, 물리적 구조를 정의)
- 컴퓨터에 오래 보관하고, 쉽게 꺼내 쓸 수 있게 만드는 것

### 예시: 카페 주문 시스템

✅ 어떤 데이터가 있는가?

- 손님(Customer) → 이름, 전화번호

- 메뉴(Menu) → 메뉴명, 가격

- 주문(Order) → 주문시간, 결제금액

✅ 서로 어떻게 연결되는가?
| 관계 | 설명 |
| ------- | --------------------------------- |
| 손님 ↔ 주문 | 한 명의 손님이 여러 번 주문할 수 있음 (1:N 관계) |
| 주문 ↔ 메뉴 | 한 번의 주문에 여러 메뉴가 포함될 수 있음 (N:M 관계) |

## DataBase Modeling 의 필요성

### 1. 데이터 무결성 유지

- 데이터의 일관성, 정확성 보장

#### 예시

| 고객명 | 전화번호         |
| ------ | ---------------- |
| 김하늘 | 010-1234-5678    |
| 김하늘 | 010-1234-5678    |
| 김하늘 | 010-9999-8888 ❌ |

-> 모델링이 잘 안돼 있으면 같은 손님인데 전화번호가 다르게 저장될 수도 있음

-> 반면, 고객 테이블을 따로 두고 고유 ID 로 관리하면 이런 오류를 막을 수 잇음

### 2. 중복 최소화

- 데이터의 중복을 방지하여 저장 공간을 절약

#### 예시

주문 테이블마다 손님 이름, 전화번호를 다 적는다면?

| 주문번호 | 고객명 | 전화번호      | 메뉴명     |
| -------- | ------ | ------------- | ---------- |
| 1        | 김하늘 | 010-1234-5678 | 아메리카노 |
| 2        | 김하늘 | 010-1234-5678 | 라떼       |

-> 같은 정보가 반복돼서 저장 공간 낭비 + 수정 시 번거로움

### 3. 효율적인 데이터 검색

최적화된 구조를 통해 빠른 검색과 성능 향상

#### 예시

- 고객, 주문, 메뉴가 따로 잘 연결되어 있다면 김하늘 고객이 주문한 메뉴 내역 같은 것도 한 번의 JOIN 으로 금방 찾을 수 있음

### 4. 유지보수 용이성

- 데이터 구조 변경시 최소한의 영향을 미침

#### 예시

카페가 포인트 적립 기능을 추가한다고 가정. 모델링이 잘 되어있다면 단순히 Point 테이블 하나 추가하고 Customer 이랑 관계 연결만 하면 끝 !

반면, 모델링이 엉망이면 모든 테이블에 포인트 컬럼을 하나씩 추가해야할지도 ...
